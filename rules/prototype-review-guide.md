# 原型 Review 指导

用于审查原型的业务完整性、项目对齐、逻辑连贯性、数据一致性、状态、异常、边界条件和恢复路径。执行前先读取 `rules/review-common-guide.md`；视觉质量按 `rules/ui-review-guide.md` 处理。

## 审查入口

当用户要求「原型评审」「需求评审」「业务逻辑评审」「看看缺漏」「检查状态和错误处理」或从业务角度挑问题时，读取本规则并输出 Markdown 评审结论，不输出 JSON。

## 审查依据

按以下优先级使用证据：

1. 用户当前说明、附件、链接、截图、需求文档、PRD、会议纪要、业务规则、字段表和数据样例。
2. 当前原型主规格：依次读取 `.spec/spec.html`、`.spec/spec.md`，同时存在时以 HTML 为准；再读取主规格链接的必要子文档。
3. `src/resources/` 中与当前原型相关的需求、业务资料、字段清单、流程记录、数据样例和 `src/resources/**/*.excalidraw` 画布。
4. 当前原型源码、`annotation-source.json`、路由、页面配置、局部状态实现和相关资源。
5. `.axhub/make/project.json` 等 metadata，仅作为资源、入口和命名关系证据。

如果用户信息与项目旧文件冲突，以用户当前信息为准，并标记为「资料冲突/待确认」，不使用旧文件覆盖用户最新资料。

## 明确不审

- 不评价视觉美观、品牌一致性、响应式布局、可访问性或是否符合 `DESIGN.md`。
- 不把配色、圆角、间距、字体或图标风格作为业务 Review 问题。
- UI 缺失直接导致业务状态、错误、流程或字段无法表达时，才记录为业务完整性问题。

## 审查流程

1. 确定原型或页面目标；目标不清时先请用户确认。
2. 按 `rules/requirements-alignment-guide.md` 的“原型主规格”确认规格；缺失时停止评审并提示创建。
3. 建立业务基线：核心用户、业务目标、主流程、关键对象、成功标准和明确范围。
4. 检查主流程入口、出口、成功路径、必要分支和空/加载/成功/失败/权限不足等状态。
5. 检查实体、字段、枚举、状态、单位、时间、金额和数量口径是否一致。
6. 检查输入边界、网络/系统/权限失败、并发或重复操作、数据过期和流程中断是否有恢复路径。
7. 写入 `src/prototypes/<prototype-id>/.spec/reviews/prototype-review.md`；页面级报告按需使用 `<page-id>-prototype-review.md`。

## 核心审查维度

- **原型完整性**：核心用户、主流程、入口、出口和关键状态是否覆盖。
- **项目上下文一致性**：是否与用户资料、最新决策、资源文档和项目命名一致。
- **业务逻辑连贯性**：流程顺序、状态迁移、角色权限、操作前置条件和数据生命周期是否自洽。
- **数据模型一致性**：实体、字段、枚举、状态、单位和业务口径是否一致。
- **边界与恢复**：输入校验、系统/权限/集成失败、并发操作和恢复路径是否完整。

## 领域评分约束

除公共规则外，应用以下封顶：

- 缺少核心用户、主流程或成功标准且只能从源码推断，最高 72。
- 关键状态、异常或恢复路径基本未覆盖，最高 75。
- 用户资料与项目资料存在未解决冲突，最高 75；冲突阻断核心目标时按 P0 处理。

## 报告结构

- 使用 `src/resources/templates/prototype-review-report-template.md`；`title` 固定为 `原型评审`。
- 公共前三组之后，第四组固定为 `完整性与项目对齐`。
- 可以追加 `业务逻辑连贯性`、`状态、异常、边界与恢复`、`证据与评估说明`。

## 优先级

- `P0`：核心业务流程无法成立，或与用户明确需求、最新决策冲突。
- `P1`：主要流程、关键状态、权限或数据口径存在严重缺漏。
- `P2`：明显业务摩擦或边界缺失，但存在可用绕行。
- `P3`：低风险补充项，完善后会提升业务表达或测试覆盖。

## 独立评估分工

需要拆分评估时使用两个互不干扰的视角：

- 业务基线评估：只看用户资料、主规格及其必要子文档、`src/resources/` 和 metadata。
- 原型证据评估：看源码、路由、状态、页面实现和相关资源。

## 交付补充

除公共交付信息外，说明使用的用户资料和项目资料，以及是否发现资料冲突或待确认项。
